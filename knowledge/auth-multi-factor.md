# Multi-Factor Authentication: TOTP, Hardware Keys, Push & MFA Fatigue

## Overview

**Multi-factor authentication (MFA)** requires the user to prove identity via two or more independent methods, reducing account takeover risk even if one factor is compromised. Common factors include: something you know (password), something you have (phone, security key), something you are (biometric). MFA implementations range from simple SMS OTP to phishing-resistant passkeys and hardware keys. However, MFA fatigue attacks exploit excessive prompts to trick users into approving unauthorized logins.

---

## MFA Factors

### Factor Categories

| Category | Examples | Security | Vulnerability |
|----------|----------|----------|---|
| **Knowledge** | Password, PIN, security question | Weak (passwords breached) | Phishing, brute force |
| **Possession** | Phone (SMS/app), hardware key | Strong (requires device) | SIM swap, theft |
| **Biometric** | Fingerprint, facial recognition | Strong (unique to user) | Spoofing (deepfake, fake print) |
| **Location** | GPS, IP geolocation | Weak (spoofable) | VPN/proxy bypass |
| **Behavioral** | Typing patterns, mouse movement | Weak (not reliable) | Difficult to spoof but noisy |

**Best combinations:**
- Password (knowledge) + TOTP (possession)
- Password (knowledge) + Hardware key (possession + biometric)
- Passkey (possession + biometric) — single factor but strong

## TOTP (Time-Based One-Time Password)

### RFC 6238 Standard

TOTP generates time-synchronized, one-time codes using user's phone, laptop, or third-party authenticator app.

### How TOTP Works

1. **Setup (Provisioning):**
   - Server generates random **secret key** (shared secret): `JBSWY3DPEBLW64TMMQ======`
   - Server encodes secret as QR code with parameters:
     ```
     otpauth://totp/user@example.com?secret=JBSWY3DPEBLW64TMMQ&issuer=Example
     ```
   - User scans QR with authenticator app (Google Authenticator, Authy, 1Password)
   - App stores secret; user is now enrolled in TOTP

2. **Authentication (Generation):**
   - Authenticator app generates code using current time + secret:
     ```
     code = HMAC-SHA1(secret, current_time_in_30sec_windows)
     → code (6-digit number): 123456
     ```
   - Code valid for 30 seconds; then next code generated
   - User enters code: `123456`

3. **Validation (Server-Side):**
   - Server knows secret; calculates expected code for current time window
   - Server also checks previous/next time window (accounts for clock skew, user delay)
   - If code matches within 30-90 second window: ✅ Approved
   - If code doesn't match: ❌ Denied

```python
import pyotp
import time

# Setup
secret = pyotp.random_base32()  # "JBSWY3DPEBLW64TMMQ======"
totp = pyotp.TOTP(secret)

# Provisioning: give user this URL
provisioning_uri = totp.provisioning_uri(name="user@example.com", issuer_name="Example")

# Authentication
user_entered_code = "123456"
is_valid = totp.verify(user_entered_code)  # Checks within ±1 time window

# Store secret in database with user account
db.save_totp_secret(user.id, secret)
```

### TOTP Advantages
- ✅ Phishing-resistant: Code bound to time; can't be reused, attacker can't intercept
- ✅ No network required: Works offline; authenticator app doesn't need internet
- ✅ Works everywhere: Compatible with standard apps (Google Authenticator, Authy)
- ✅ Cheap: Only need a phone; no hardware purchase

### TOTP Disadvantages
- ❌ No possession proof: Any code generated by app registers as valid
- ❌ User responsibility: If device lost, codes inaccessible
- ❌ Screen requirement: User must look at phone to find code
- ❌ UX friction: User copy-paste or re-type code

## HOTP (HMAC-Based OTP)

### RFC 4226 Standard

HOTP generates counter-based one-time codes (older than TOTP; less common).

```
code = HMAC-SHA1(secret, counter)
```

- **Counter** increments with each code generation (not time-based)
- User clicks button on device to generate next code
- Attacker can't predict next code without possession

**Use cases:**
- Hardware tokens (RSA SecurID, Yubikey)
- Situations where time sync unreliable

**Difference from TOTP:** Counter-based vs time-based; TOTP more common now.

## SMS and Voice OTP

### How It Works

1. Server generates random code: `123456`
2. Server sends code via SMS or voice call
3. User receives on phone; enters code

### Vulnerabilities

**SIM Swap Attack:**
- Attacker calls mobile carrier, claims to be user
- Carrier transfers phone number to attacker's SIM
- Attacker receives SMS codes; gains account access
- User's phone stops receiving SMS (SIM slot empty or swapped)

**Real example:** Hacker gains access to Twitter account (@jack) via SIM swap.

**Solution:**
- Carrier: Require multi-factor authorization (PIN, ID, security questions)
- User: Ask carrier to PIN phone number (extra protection)
- Service: Don't rely solely on SMS; offer strong alternatives

**Compromise Attack (Telecom Breaches):**
- Attacker compromises telecom infrastructure
- Intercepts SMS in transit
- Decrypts code

**Solution:**
- Use TOTP or passkeys instead of SMS
- If SMS required: Combine with other factors

**Recommendation:** ⚠️ **Avoid SMS if possible.** Use TOTP, passkeys, or hardware keys instead. If SMS needed: combine with other factors, PIN protect phone number.

## Push Notifications

### How It Works

1. User logs in with password
2. Server sends push notification to user's phone: "Login attempt from desktop?"
3. User sees prompt on phone
4. User taps "Approve" or "Deny"
5. Phone sends response back to server
6. If approved: desktop login completes

### Advantages
- ✅ Intuitive: User sees context ("login from New York"; "at 3am")
- ✅ No friction: No copy-paste; just tap
- ✅ Phishing-resistant: User approves specific login attempt
- ✅ Flexible: Can ask for additional context

### Disadvantages
- ❌ Network required: Push needs internet and app
- ❌ Dependency on phone: If phone battery dead, can't approve
- ❌ Delay: Slight latency between request and approval
- ❌ Push fatigue: Too many prompts → users approve without thinking (next section)

### Recommended Platforms

- **Google Authenticator:** Push via Google account
- **Microsoft Authenticator:** Push via Microsoft account
- **Okta Verify:** Push for enterprise
- **Duo Security:** Push for MFA (now owned by Cisco)

## Hardware Keys (FIDO U2F, FIDO2)

### FIDO U2F (Universal Second Factor)

- Physical security key (USB dongle, NFC, Bluetooth)
- User plugs in key, presses button
- Key signs authentication challenge with private key
- Phishing-resistant; can't be intercepted or replayed

### FIDO2 / WebAuthn

Modern standard; superset of U2F. Also supports passkeys (see [auth-passkeys.md](auth-passkeys.md)).

### Advantages
- ✅ Strongest auth: Private key never leaves device; phishing-resistant
- ✅ Works everywhere: USB-A, USB-C, NFC, Bluetooth
- ✅ No codes to type
- ✅ Tamper-evident: Key is physical; theft is obvious

### Disadvantages
- ❌ Cost: $20-100 per key
- ❌ Distribution: Enterprises must ship keys
- ❌ Recovery: If key lost, account locked (need backup key or recovery codes)
- ❌ Setup friction: Company buy + ship + distribute keys

### Use Cases
- High-security accounts (executives, sensitive roles)
- High-risk users (journalists, activists, political figures)

## Biometric Authentication

### Methods

| Method | How | Spoofing Risk | Speed |
|--------|-----|---|---|
| **Fingerprint** | Scan finger on sensor | Medium (fake prints, lifted latent prints) | Fast (< 1s) |
| **Facial Recognition** | Camera captures face | Medium-High (photos, deepfakes, mask attacks) | Fast (< 1s) |
| **Voice Recognition** | Identify by voice | Medium (recording playback, voice synthesis) | Slower (1-3s) |
| **Iris Scan** | Scan eye iris | Low (difficult to spoof) | Medium (2-5s) |
| **Gait Recognition** | Identify by walking pattern | Medium (hard to replicate, vary with fatigue/shoes) | Slower (5-10s) |

### MFA Use Cases

Biometrics typically paired with password or hardware key:

```
1. User enters password
2. Server sends push notification to phone
3. User approves via phone fingerprint
4. Login completes
```

Or on mobile:
```
1. User accesses app
2. App shows login
3. User uses phone's Face ID / Touch ID
4. App authenticated; user logged in
```

### Vulnerabilities

**Spoofing:**
- High-quality photo can bypass facial recognition
- Deepfake video can impersonate voice
- Silicone finger can bypass fingerprint sensor

**Defense:**
- Liveness detection: Require blink, smile, turn head (video, not photo)
- Multi-spectral scanning (reads blood flow under skin, defeats latex prints)
- Anti-replay: Challenge user to perform action (can't replay recording)

## MFA Fatigue Attacks

### The Problem

User registers multiple MFA methods (phone push, TOTP, security key). Attacker compromises password and begins MFA prompts:

```
Attacker tries login → Phone: "Approve?"
User denies → Attacker tries again
→ Phone: "Approve?" (again within minutes)
→ Phone: "Approve?" (repeated prompts)
User: "Stop the notifications; I'll just approve"
User approves attacker's login
```

**Real case:** Uber breach (2022) — attacker compromised employee password, spammed MFA prompts, employee approved out of fatigue.

### Defense Against MFA Fatigue

**Rate Limiting (per-user MFA attempts):**
```python
# Track failed MFA attempts
def increment_mfa_failures(user_id):
    count = redis.incr(f"mfa_failures:{user_id}")
    if count > 5:  # 5 failed attempts
        redis.setex(f"mfa_lockout:{user_id}", 15*60, "1")  # 15-minute lockout
        return False
    return True
```

**Risk-Based Prompting:**
- Familiar device/location → may skip MFA
- New device/location → require MFA
- Impossible travel → require MFA + manual review

**Contextual Alerts:**
- User phone displays: "Unusual login: New York, 3am, Windows; approve?"
- User can see context and decide if legitimate
- Attacker can't control what's shown

**Approval Windows:**
- MFA approval valid only for 2-5 minutes
- Attacker can't sit on approved login indefinitely
- User sees: "Approved for 2 minutes; use immediately"

**Multiple Approval Challenge:**
- Instead of one "Approve/Deny" prompt
- Require user to select matching code or answer security question
- Prevents accidental approval

## Adaptive and Step-Up Authentication

### Risk-Based Adaptive Auth

System dynamically adjusts MFA requirements based on risk signals:

```
Low-risk login (home device, familiar location, normal time):
  → Password only (skip MFA)

Medium-risk (new location, after hours):
  → Password + TOTP

High-risk (new device, impossible travel, multiple failed logins):
  → Password + hardware key + manual review
```

**Risk signals to track:**
- Device: Is this device known to user?
- Location: Is geolocation plausible? Impossible travel?
- Time: Is login during user's normal hours?
- Behavior: Is request pattern consistent with user's history?
- Velocity: Multiple login attempts in short time?
- Network: VPN, Tor, known datacenter, carrier network?

**Implementation:** Machine learning classifier or rules engine.

### Step-Up Authentication

Increase authentication strength when accessing sensitive resources:

```
User logs in with password → Browsing normal data
  → No additional auth required

User clicks "Download Private Documents":
  → System prompts: "Confirm with fingerprint"
  → User provides biometric
  → Download approved
```

Prevents:
- Session hijacking (attacker steals session, tries sensitive action, gets blocked)
- Unaware user accidentally exposing sensitive data

## MFA Adoption and Recovery

### Recovery Codes

User generates backup single-use codes during MFA setup:

```
MFA Setup → Show user 10 recovery codes
User stores codes (printed, screenshot, password manager)
If phone lost: User enters recovery code instead of TOTP
Code consumed; cannot be reused
```

**Best practice:** User generates new recovery codes after use.

### Backup Methods

User registers multiple MFA methods:

```
Primary: TOTP on phone
Secondary: Backup security key
Tertiary: Recovery codes
```

If phone lost:
- Use security key (skips TOTP)
- If key also lost: Use recovery codes

### Account Recovery Flow

User lost all MFA factors:

```
User: "I lost my phone and security key"
System: "Verify identity via security questions / email verification"
User: Passes verification
System: "Your account is temporarily locked for 24 hours"
  (Prevents account takeover with stolen password)
System: "After 24 hours, contact support to reset MFA"
  (Human review; verify user identity again)
User: Re-enrolls in MFA with new device
```

## MFA Implementation Patterns

### For Web Applications

```javascript
// After password validation
const mfaMethods = await user.getMFAMethods();
return { mfaRequired: true, methods: ["totp", "push", "backup_codes"] };

// User selects method
if (method === "totp") {
  const userCode = await prompt("Enter 6-digit code");
  const isValid = await verifyTOTP(userCode, user.totpSecret);
  if (isValid) createSession(user);
}

if (method === "push") {
  await sendPushNotification(user.device, "Approve login?");
  const approved = await waitForPushResponse(5000);  // 5 second timeout
  if (approved) createSession(user);
}
```

### For API and M2M

MFA less common for service-to-service auth. Alternative: mutual TLS (mTLS), API keys with IP whitelisting, OAuth with short-lived tokens.

If MFA required for API:
- Issue MFA token on successful MFA verify
- API requests must include MFA token
- Token expires quickly (5-15 minutes)

## MFA Security Checklist

- ✅ Require password + at least one strong factor (TOTP, hardware key, biometric)
- ✅ Offer multiple MFA methods (user choice)
- ✅ Rate-limit MFA attempts (5 failures → 15-minute lockout)
- ✅ Require backup methods (recovery codes, secondary device)
- ✅ Use phishing-resistant methods (passkeys, hardware keys, TOTP)
- ✅ Avoid SMS if possible; require with other factors if unavoidable
- ✅ Implement account recovery (delayed, requires verification)
- ✅ Consider risk-based adaptive auth (skip MFA for low-risk, require for high-risk)
- ✅ Protect against MFA fatigue (rate-limit, contextual prompts, approval windows)
- ✅ Audit MFA usage (log enrollments, successful/failed attempts)
