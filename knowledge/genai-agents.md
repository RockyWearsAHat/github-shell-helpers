# AI Agent Architectures — Reasoning, Planning & Tool Use

## Agent Fundamentals

An AI agent is an LLM-based system that moves beyond simple text generation to autonomous action: it observes the environment (via context), reasons about next steps, and invokes tools to change state. The agent loop typically looks like:

```
Observation (context, tool outputs)
  ↓
Thought (internal reasoning; generated by LLM)
  ↓
Action (tool invocation or final response)
  ↓
Observation update
  ↓ (repeat until done or max steps)
Final Answer
```

The key capability is **agentic loop closure**: agents must detect when they've gathered enough information to answer without further action. Agents that call tools infinitely or produce incorrect final answers are failures, even if individual steps are reasonable.

## ReAct: Reasoning and Acting

ReAct (Yao et al., 2022) combines chain-of-thought reasoning with tool invocation. The agent alternates between:

- **Thoughts**: Explicit step-by-step reasoning (e.g., "I need to search for the population of France to answer this")
- **Actions**: Tool calls with structured output (e.g., `search("population of France")`)
- **Observations**: Tool results, fed back into context

ReAct is elegantly simple: it works by asking the LLM to generate both reasoning and tool calls in a predictable format, then parsing the output to extract and execute tools. This works because modern LLMs learn this pattern readily across a wide range of tasks.

| Advantage | Trade-off |
|-----------|-----------|
| Transparent reasoning; easy to debug | Verbose; token overhead per step |
| Works with any LLM capable of function calling | Hallucinations (fake tool calls); needs guardrails |
| Modular tool set; new tools added without retraining | Poor at complex multi-step planning |

## Function Calling and Tool Use Protocols

Modern LLMs have native support for **function calling**: the model outputs structured calls to tools rather than freeform text. APIs provide tool schemas as JSON:

```json
{
  "type": "function",
  "function": {
    "name": "search",
    "parameters": {
      "type": "object",
      "properties": {
        "query": {"type": "string"},
        "max_results": {"type": "integer"}
      },
      "required": ["query"]
    }
  }
}
```

The LLM learns to respond with structured tool calls; the system parses these and executes them. This is more reliable than regex-parsing freeform agent output and enables the LLM to reason over available tools when deciding what to call.

Protocol standardization efforts (OpenAI, Anthropic, Google) mean tool definitions are increasingly portable across providers. However, different models may order tool calls differently or misinterpret ambiguous schemas.

## Planning Strategies: CoT, ToT, and GoT

**Chain-of-Thought (CoT)**: Linear reasoning; the agent generates one next-step decision at a time. Works well for straightforward problems but poor for domains where backtracking is necessary.

**Tree of Thoughts (ToT)**: Explores multiple reasoning branches in parallel. The agent generates several candidate next steps, evaluates each, and prunes low-probability branches. More robust but expensive (more LLM calls).

```
              [Problem]
              /    |    \
          Step1A  Step1B  Step1C (evaluate each)
          /   \    |      /   \
       Step2A Step2B ...        (self-evaluate, prune, continue)
```

ToT works well for puzzles, mathematical reasoning, and complex planning. Cost is linear in the branching factor—exploring 3 candidates per step in a 5-step problem means 3^5=243 LLM calls. Practical implementations limit branching.

**Graph of Thoughts (GoT)**: Generalizes ToT by allowing arbitrary edges between reasoning nodes—not just a tree. Enables revisiting and combining earlier insights. More flexible but requires explicit graph management and evaluation metrics to decide which paths to explore.

## Memory Systems

Agents operating over long horizons need persistent memory; otherwise, they forget previous decisions and context after a few exchanges.

**Short-term memory**: The current prompt context (typically the last N messages + tool results). Usually 4K-32K tokens. Fast and cheap to query but limited capacity.

**Long-term memory** has multiple implementations:

- **Retrieval-augmented**: Store past interactions as embeddings; retrieve relevant ones for each step. Scales to millions of interactions but requires embedding infrastructure.
- **Summarization-based**: Periodically summarize long conversation histories into bullet points; prepend summary to context. Simple but loses fine-grained details.
- **Learned state**: Train a separate model on agent trajectory to compress history into a fixed latent vector. Complex but theoretically most compact.

Practical agents mix all three: a small summary of conversational context + retrieval over key facts + current conversation. The mixture depends on whether accuracy (retrieval) or cost (summarization) is prioritized.

## Multi-Agent Coordination

When multiple agents solve different parts of a problem (e.g., one agent researches, another writes, a third reviews), coordination overhead increases. Key challenges:

- **Task decomposition**: How to split the problem so agents don't duplicate effort
- **Information sharing**: Passing state between agents (shared memory, message passing, broadcast)
- **Consensus and conflict**: Agents may produce contradictory outputs; resolving requires explicit mediation
- **Load balancing**: Some tasks finish faster than others; queueing becomes necessary

Simple approaches (sequential chaining: agent A → output → agent B) work well when task dependencies are linear. DAG-based orchestration (e.g., LangGraph) allows more complex dependencies but requires explicit coordination logic.

## Safety and Guardrails

Autonomous agents can cause harm if they:
- Execute dangerous tool calls (delete files, send unintended messages)
- Hallucinate tool outputs, then act on them (agent believes something false and continues)
- Enter infinite loops (tool calls never converge to a final answer)
- Reject legitimate requests due to misaligned guardrails

Common countermeasures:
- **Tool input validation**: Restrict callable functions, parameters, and argument ranges
- **Output filtering**: Reject tool calls that match unsafe patterns
- **Step limits**: Enforce maximum iterations; force a final answer
- **Human-in-the-loop**: For high-stakes decisions, require approval before executing
- **Explainability**: Log reasoning traces to audit why agents made decisions

No mechanism is foolproof; adversarial users can often craft prompts that bypass checks. Defense-in-depth (multiple layers) is standard practice.

## Evaluation Challenges

Evaluating agents is harder than evaluating models because:
- **Trajectory variance**: Same agent with same input may take different paths
- **Multi-step errors**: An error early in the trajectory compounds downstream
- **Goal definition**: "Solve the task" is ambiguous—what counts as success?

Practical metrics:
- **Success rate**: Fraction of tasks where final answer is correct
- **Tool call accuracy**: Did the agent invoke the right tools with sensible arguments?
- **Efficiency**: Steps to success (fewer is better; agents with longer chains are less reliable)
- **Human evaluation**: For open-ended tasks (writing, creative problem-solving), human raters assess output quality

Current systems rarely exceed 70-85% success on complex multi-step reasoning tasks. Scaling agents (more tokens, more search, test-time compute) improves success but at cost.
